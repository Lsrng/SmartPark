# 物联设备数据上报接口 — 限流方案设计文档

---

## 一、需求描述

### 1.1 背景
智慧园区系统中，物联设备（如温湿度传感器等）会定期上报设备状态和监测数据。正常情况下，每个设备**每分钟上报一次**数据。

### 1.2 问题
由于物联设备可能出现故障（如程序 bug、网络重连、传感器抖动等），导致设备**在短时间内频繁上报数据**，大量请求直接打到数据库，可能引发以下问题：
- 数据库连接池耗尽
- 数据库写入吞吐量瓶颈
- 服务响应变慢甚至雪崩

### 1.3 需求
实现一个物联设备数据上报接口，要求：
- 接收物联设备上报的数据并写入数据库
- 对每个设备进行**限流（Rate Limiting）**，阈值为 **每分钟 1 次**
- 超限的请求应被拒绝，并返回明确的错误提示
- 方案应具备一定的**健壮性和可扩展性**

---

## 二、解决思路

### 2.1 核心逻辑
限流的本质是**在单位时间内限制某个维度的请求次数**。

针对"每分钟 1 次"这个阈值，每个设备独立限流。核心判断逻辑为：

> 对于设备 `A`，判断在最近 60 秒内是否已经成功上报过数据。
> - 如果**已经上报过** → 拒绝本次请求
> - 如果**没有上报过** → 允许本次请求，并记录上报时间

### 2.2 关键维度
限流的几个关键维度如下：

- **限流粒度**：按设备 ID 隔离，设备之间互不影响，每个设备拥有独立的计数器和时间窗口。
- **时间窗口**：60 秒。可以采用固定窗口（整分钟对齐）或滑动窗口（以当前时间为基准向前推 60 秒）。
- **阈值**：每个设备每分钟 1 次，即单位窗口内只允许 1 次成功上报。
- **超出处理**：拒绝请求，返回 HTTP 429 Too Many Requests 状态码，配合友好提示信息告知客户端。

### 2.3 需要考虑的点
- **并发安全**：多线程/多实例场景下不能出现超放行，需要保证限流判断和计数更新的原子性。
- **资源清理**：长时间不活动的设备数据不应常驻内存或 Redis，避免资源浪费。
- **重启恢复**：服务重启或崩溃后，限流状态不应丢失或出现混乱，避免设备在重启后瞬间涌入大量请求。
- **未来扩展**：如果后续扩展为多实例集群，方案是否可以直接支持，还是需要大规模重构。

## 三、解决方案

### 方案 A：基于数据库查询 + 时间判断

#### 方案思路
在数据库 `device_data_record` 表中已有设备上报记录，每次请求到达时，查询该设备最近一次上报记录的时间，判断是否超过 60 秒阈值。

1. 请求到达时，根据设备 ID 查询数据库，取该设备最近一条上报记录的 `report_time`
2. 如果查询到记录，且当前时间 - 上次上报时间 < 60 秒，则拒绝请求
3. 否则将数据写入数据库，放行

#### 技术选型
- `MySQL` + `MyBatis-Plus` + `Spring AOP` 自定义注解
- 查询最近一条记录：`SELECT report_time FROM device_data_record WHERE device_id = ? ORDER BY report_time DESC LIMIT 1`
- 零额外依赖，复用项目已有的数据库和 MyBatis-Plus

#### 优点
- **实现极简**：直接复用现有数据库，不需引入任何新组件或中间件，在 Service 层通过一条 SQL 查询即可实现。
- **持久化不丢失**：上报时间存储在数据库中，服务重启、发版部署、崩溃恢复后限流状态依然存在，设备仍需等待 60 秒才能再次上报。
- **天然分布式**：多实例共享同一数据库，限流状态全局一致，扩展为零成本。
- **与业务数据一致**：限流判断与业务数据在同一数据库中，无需同步，无数据一致性问题。

#### 缺点
- **与保护数据库的初衷相悖**：限流的目的是为了保护数据库，但该方案每来一次请求就需要查一次数据库，在设备大量频繁上报时反而会加剧数据库压力，治标不治本。
- **并发安全性差**："查询上次时间 → 判断是否超限 → 插入新记录"这三个操作不是原子的。高并发下，多个请求同时查询到「没有最近记录」，然后同时插入，导致多个请求都通过限流，出现超放行。需要通过数据库锁（如 `SELECT ... FOR UPDATE`）或唯一约束来保证，增加复杂度。
- **性能瓶颈**：每次请求都走一次数据库查询，查询延迟在毫秒到十毫秒级别，远高于内存或 Redis 方案。在设备数量多、并发高的场景下，数据库本身可能成为瓶颈。
- **数据库连接压力**：限流判断占用数据库连接，在流量高峰时可能加剧连接池耗尽的风险，与限流保护数据库的目标背道而驰。
- **资源浪费**：仅为了限流判断就发起一次数据库查询，而大部分请求在正常场景下是被放行的，最终还要再执行一次 insert 写库，相当于一次请求至少两次数据库操作。

---

### 方案 B：本地内存 + 固定窗口

#### 方案思路
在应用内存中维护一个 `ConcurrentHashMap<设备ID, 上次上报时间戳>`。请求到达时：

1. 从 Map 中取出该设备的上次上报时间戳
2. 如果当前时间 - 上次上报时间 < 60 秒，则拒绝请求
3. 否则更新 Map 中的时间戳，放行请求
4. 定时任务定期清理 Map 中长时间未活动的设备记录

#### 技术选型
- `ConcurrentHashMap` + `Spring AOP` + 自定义注解
- 零额外依赖（项目已有 Spring AOP 和 Hutool）

#### 优点
- **实现简单**：1 个切面类 + 1 个注解即可完成，代码量极少，开发效率高。
- **性能极高**：纯内存操作，纳秒级响应，无网络 IO 开销。
- **零依赖**：无需引入任何第三方中间件，项目已有的 Spring AOP 即可满足。
- **部署轻量**：开箱即用，无额外运维成本，不依赖外部服务。

#### 缺点
- **不支持分布式**：多实例部署时各节点状态独立，无法共享限流数据。如果后续扩展为集群，该方案需要整体重构。
- **重启丢失**：JVM 重启后限流计数归零，设备可能瞬间涌入大量请求，失去限流保护。
- **内存占用**：设备量达到数十万级别时，Map 的内存占用不可忽略，需要考虑内存上限和 GC 压力。
- **固定窗口边界问题**：固定 60 秒窗口在边界处可能出现"双倍请求"，例如在 59.9 秒和 60.1 秒各通过一次，实际上不到 1 秒内通过了 2 次请求。
- **并发问题**：高并发下需使用 `computeIfAbsent` 等原子操作，否则可能出现多个线程同时判断为"未限流"而超放行的情况。

---

### 方案 C：Redisson RRateLimiter

#### 方案思路
使用 Redisson 提供的 **RRateLimiter** 分布式限流器，基于令牌桶算法实现。

1. 为每个设备创建一个限流器 key：`rate_limiter:device:{deviceId}`
2. 设置限流器速率：`setRate(RateType.PER_CLIENT, 1, 60, RateIntervalUnit.SECONDS)`
3. 每上报请求到达时，调用 `tryAcquire()`：
   - 返回 `true` → 获取到令牌，继续执行上报逻辑
   - 返回 `false` → 限流，拒绝请求
4. 限流器在 Redis 中自动管理，无需手动清理

#### 技术选型
- `Redisson` + `Spring Boot` + `AOP`（自定义注解）
- 项目 pom.xml 中已有 Redisson 依赖（被注释），启用即可

#### RRateLimiter 核心 API 说明
```java
// 创建或获取限流器
RRateLimiter limiter = redissonClient.getRateLimiter("rate_limiter:device:001");

// 设置限流规则：每 60 秒最多产生 1 个令牌
limiter.trySetRate(RateType.PER_CLIENT, 1, 60, RateIntervalUnit.SECONDS);

// 尝试获取 1 个令牌，立即返回结果
boolean acquired = limiter.tryAcquire();
```

#### 优点
- **Redisson 封装完善**：开箱即用的分布式限流器，API 简洁优雅，一行代码即可完成限流判断。
- **令牌桶算法平滑**：支持一定突发能力，比固定窗口更平滑。例如设备偶发中断后恢复，不会因为严格的固定窗口而拒绝合法请求。
- **天然分布式**：多实例共享 Redis 限流状态，未来扩展为零成本，无需任何架构调整。
- **原子安全**：Redisson 内部通过 Lua 脚本保证原子性，开发者无需关注并发安全问题。
- **自动资源管理**：限流器 Key 会自动过期释放，无需手动清理或额外编写定时任务。
- **与项目兼容**：pom.xml 中已有 Redisson 依赖，取消注释即可使用。

#### 缺点
- **需启用 Redisson 依赖**：需要在 pom.xml 中取消注释并添加 Redis 连接配置。
- **网络开销**：每次请求多一次 Redis 网络往返（约 1-5ms），相比于纯内存方案有一定延迟。
- **运维成本**：需要维护 Redis 服务的可用性和性能。
- **API 耦合**：业务代码与 Redisson 框架有一定耦合，但可通过 AOP 注解方式将限流逻辑与业务逻辑解耦。
- **令牌桶行为**：Redisson RRateLimiter 基于令牌桶实现，如果 bucket 容量配置大于 1，间隔时间略长于 60 秒时可能会累积多个令牌，导致短时突发。需要确认配置中 bucket 容量为 1 以避免该问题。

## 四、方案对比

在实现难度方面，方案 B（本地内存）最为简单，只需一个 HashMap 加一个切面即可，开发工作量最小。方案 A（数据库方案）次之，只需一条 SQL 查询语句即可实现。方案 C（Redisson RRateLimiter）利用 Redisson 现成的 API，主要工作量在注解和切面封装上，同样较为简单，但需要额外配置 Redis 连接。

在性能方面，方案 B 是纯内存操作，性能极高，无额外的网络 IO。方案 C 需要一次 Redis 网络往返，带来约 1-5ms 的额外延迟，但考虑到设备上报本身是分钟级频率，这点延迟完全可以接受。方案 A 需要一次数据库查询，延迟在 1-10ms 级别，且在高并发下数据库压力最大。

在限流精准度方面，方案 C（令牌桶算法）最为平滑，且通过合理配置 bucket 容量为 1 可以避免突发问题。方案 B（固定窗口）存在边界处的"双倍请求"问题，精准度次之。方案 A 的精准度取决于数据库查询的精度和并发控制，存在超放行风险，精准度最差。

在分布式支持方面，方案 A 天然支持分布式（多实例共享同一数据库）。方案 C 也天然支持分布式（多实例共享 Redis）。方案 B 完全不支持分布式，多实例部署时各节点独立计数，会导致限流失效。

在重启安全性方面，方案 A 的数据存储在数据库中，重启后状态不丢失。方案 C 存储在 Redis 中，重启后状态也不丢失。方案 B 是内存存储，重启后限流状态丢失。

在并发安全方面，方案 A 的"查询-判断-插入"操作非原子，高并发下容易超放行，需要额外加锁。方案 B 需要谨慎使用 ConcurrentHashMap 的原子方法，否则也可能超放行。方案 C 由 Redisson 内部通过 Lua 脚本保证原子性，开发者无需关注并发安全。

在依赖要求方面，方案 A 和方案 B 都无需额外依赖，方案 C 需要启用 Redisson（项目已有注释的依赖，取消注释即可）。

在资源自动清理方面，方案 B 需要额外编写定时任务来清理长时间不活动的设备记录。方案 A 和方案 C 分别依赖数据库和 Redisson 的自动管理机制，无需额外编码。

在未来扩展性方面，方案 B 扩展为集群时需要重构为分布式方案。方案 A 和方案 C 天然支持分布式，可以直接平滑扩展。但方案 A 在设备量和请求量增长时，对数据库的压力会线性增长，扩展性受限于数据库性能。

## 五、最终方案：Redisson RRateLimiter 实现设计

### 5.1 选型理由
选择 Redisson RRateLimiter 而非其他两个方案的原因，通过与各方案的逐一对比来说明：

**与方案 A（数据库方案）对比：**

选择 Redisson RRateLimiter 而非方案 A，主要因为以下几个决定性因素。

在与保护数据库的初衷一致性方面，方案 A 每次请求都需要查询一次数据库来获取设备上次上报时间，在设备大量频繁上报时不仅没有保护数据库，反而额外增加了数据库的读取压力。而 Redisson RRateLimiter 的限流判断完全在 Redis 中完成，数据库仅在被放行的正常请求中执行写入操作，真正做到了保护数据库。

在并发安全性方面，方案 A 的"查询上次时间 → 判断是否超限 → 插入新记录"三个操作不是原子的。高并发下，多个请求同时查询到"没有最近记录"时，会同时插入多条记录，导致多个请求都通过限流。虽然可以通过 `SELECT ... FOR UPDATE` 加行级锁或唯一约束来解决，但这会增加复杂度和数据库锁竞争。Redisson RRateLimiter 内部通过 Lua 脚本保证 `tryAcquire()` 的原子性，不存在超放行问题。

在性能方面，方案 A 每次请求至少一次数据库查询（毫秒级延迟），高并发下数据库成为瓶颈。Redisson RRateLimiter 在 Redis 中完成判断（微秒级延迟），性能更高且不影响业务数据库。

在资源浪费方面，方案 A 仅为了限流判断就发起一次数据库查询，而正常请求最终还要再执行一次 insert 写库，相当于一次请求至少两次数据库操作。Redisson RRateLimiter 的限流判断与业务数据库操作解耦，互不影响。

**与方案 B（本地内存 + 固定窗口）对比：**

选择 Redisson RRateLimiter 而非方案 B，主要因为以下几个决定性因素。

在分布式支持方面，方案 B 使用 `ConcurrentHashMap` 存储在 JVM 本地内存中，所有限流状态绑定在单个进程内。如果后续业务增长需要扩展为多实例集群，方案 B 必须整体重构，各节点之间无法共享限流状态，会导致限流失效。而 Redisson RRateLimiter 基于 Redis 存储，天然支持多实例共享。

在重启安全性方面，方案 B 的限流状态随 JVM 生命周期而存在。一旦服务重启、崩溃或发版部署，所有设备的限流计数归零，设备会瞬间涌入大量请求，可能再次打垮数据库。Redisson RRateLimiter 的状态持久化在 Redis 中，重启后限流屏障依然生效。

在并发安全方面，方案 B 需要开发者自行处理高并发下的原子性问题。虽然 `ConcurrentHashMap` 提供了线程安全的数据结构，但"判断是否限流 + 更新时间戳"这一复合操作并非原子，需要额外使用 `computeIfAbsent` 等 CAS 操作来保证，稍有不慎就会出现超放行。Redisson RRateLimiter 内部通过 Lua 脚本保证了 `tryAcquire()` 的原子性，开发者无需关心底层并发细节。

在资源清理方面，方案 B 需要额外编写 `@Scheduled` 定时任务来清理长时间不活动的设备记录，否则 Map 会无限制增长，可能导致内存泄漏。Redisson RRateLimiter 的 Key 会自动过期释放，无需任何手动维护。

**综合结论：**

基于以上对比，选择 Redisson RRateLimiter 的综合理由可归纳为：它克服了方案 A 与保护数据库初衷相悖、并发安全性差、性能瓶颈等固有缺陷；又弥补了方案 B 不支持分布式、重启丢失状态、需要自行处理并发安全和资源清理等问题。同时项目 pom.xml 中已有 Redisson 依赖（仅需取消注释），引入成本极低。在当前有 Redis 基础设施的前提下，Redisson RRateLimiter 是最优选择。

### 5.2 实现架构

```
┌─────────────────────────────────────────────┐
│               Controller 层                  │
│        DeviceDataController                 │
│         @PostMapping("/device/data")        │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────▼──────────────────────────┐
│             Service 层 (业务逻辑)             │
│       DeviceDataService.report(data)        │
└──────────────────┬──────────────────────────┘
                   │ @RateLimit(interval = 60)
┌──────────────────▼──────────────────────────┐
│         AOP 切面层 (RateLimitAspect)         │
│    redissonClient.getRateLimiter(key)        │
│         limiter.tryAcquire()                 │
│   ┌──────┴──────┐                           │
│   │ true        │ false                      │
│   │ (放行)      │ (拒绝)                     │
│   └──────┬──────┘                           │
│          │                                   │
│   ┌──────▼──────┐                           │
│   │ 执行原方法  │ 抛出 RateLimitException    │
│   └─────────────┘                           │
└─────────────────────────────────────────────┘
```

整个调用链路为：Controller 接收设备上报请求 → 调用 Service 层的上报方法 → AOP 切面拦截带有 `@RateLimit` 注解的方法 → 通过 Redisson RRateLimiter 判断是否限流 → 放行则执行业务逻辑写库，限流则抛出异常由全局处理器统一处理。

### 5.4 各组件职责

- **`@RateLimit` 注解**：定义限流维度（如按设备 ID）、限流阈值（每分钟 1 次）。标注在需要进行限流的 Service 方法上。
- **`RateLimitAspect` 切面**：拦截所有带 `@RateLimit` 注解的方法。从方法参数中提取设备 ID，构造 Redis key，获取对应的 RRateLimiter 并调用 `tryAcquire()` 进行限流判断。
- **`RateLimitException` 异常**：自定义限流异常类，包含被限流的设备 ID、限流阈值等信息，便于全局异常处理器构造错误响应。
- **`GlobalExceptionHandler`**：在现有的全局异常处理器中新增对 `RateLimitException` 的捕获处理，返回 HTTP 429 状态码及友好的错误提示。
- **`DeviceDataController` 控制器**：接收设备上报的 HTTP 请求，校验请求参数，调用 Service 层处理业务。

## 六、兜底方案：Guava RateLimiter（本地限流器）

### 6.1 方案背景

Redisson RRateLimiter 依赖 Redis 服务，当 Redis 宕机或网络出现异常时，限流功能将不可用。在"7.3 故障场景"中提到，Redis 不可用时有两种降级策略：放行所有请求（牺牲限流保可用）或拒绝所有请求（保限流但影响业务）。这两种策略都不理想：

- **放行所有请求**：设备故障时大量请求打到数据库，失去了限流的意义。
- **拒绝所有请求**：即使 Redis 短暂不可用，也会导致所有设备上报被拒，影响正常业务。

因此需要一个**本地兜底限流方案**：当 Redis 不可用时，自动降级为本地限流，保证限流功能不中断。

### 6.2 方案思路

使用 Google Guava 提供的 **RateLimiter**（基于令牌桶算法实现），在 Redis 异常时作为降级方案。

核心逻辑分为两层：

1. **优先使用 Redisson RRateLimiter**（分布式限流，首选）
2. **捕获 Redis/Redisson 异常时，降级为 Guava RateLimiter**（本地限流，兜底）

降级流程如下：

```
请求到达 → AOP 切面拦截
    │
    ├── try Redisson RRateLimiter
    │       ├── 成功 → 正常限流判断
    │       └── 抛出异常（Redis 不可用）
    │               ↓
    └── catch 异常 → 降级为 Guava RateLimiter
            ├── 获取令牌 → 放行
            └── 未获取到 → 拒绝（限流）
```

### 6.3 Guava RateLimiter 核心特性

- **令牌桶算法**：与 Redisson RRateLimiter 算法一致，限流行为保持一致，不会出现两种算法表现差异导致用户困惑。
- **平滑突发**（SmoothBursty）：默认模式，允许一定程度的突发流量。适合设备偶发中断后恢复的场景。
- **预热模式**（SmoothWarmingUp）：逐步达到限流速率，适合需要平滑启动的场景。
- **同步阻塞 vs 非阻塞**：`acquire()` 会阻塞等待直到获取到令牌；`tryAcquire()` 立即返回结果，更适合接口限流场景。

### 6.4 核心 API 说明

```java
// 创建限流器：每秒生成 1/60 个令牌（即每分钟 1 个）
RateLimiter limiter = RateLimiter.create(1.0 / 60);

// 非阻塞获取 1 个令牌，立即返回
boolean acquired = limiter.tryAcquire();
```

### 6.5 技术选型

- `com.google.guava:guava:32.1.3-jre`（需在 pom.xml 中新增依赖）
- `ConcurrentHashMap<设备ID, RateLimiter>` 缓存每个设备对应的限流器实例
- 需要 Guava 版本 30.0+（支持 `RateLimiter`）

### 6.6 实现要点

**Guava RateLimiter 实例管理：**

- 使用 `ConcurrentHashMap<String, RateLimiter>` 缓存每个设备的限流器实例。
- 首次获取时调用 `RateLimiter.create(1.0 / 60)` 创建新的限流器实例，通过 `computeIfAbsent` 保证线程安全。
- Map 的 key 与 Redisson 的 key 保持一致（按设备 ID），方便统一管理，降级切换对业务透明。
- 通过 `@Scheduled(fixedRate = 600_000)` 定时任务，每 10 分钟执行一次 `limiterMap.clear()` 清理所有限流器实例。如果设备后续还有上报请求，会在 `getLimiter()` 时重新创建。这种"全量清理 + 按需重建"的方式实现简单，且能有效防止内存泄漏。

**与现有 AOP 切面集成：**

在现有的 `RateLimitAspect` 中添加 fallback 逻辑：

```java
try {
    // 1. 优先使用 Redisson（分布式限流）
    RRateLimiter limiter = redissonClient.getRateLimiter(key);
    limiter.trySetRate(RateType.PER_CLIENT, 1, 60, RateIntervalUnit.SECONDS);
    if (limiter.tryAcquire()) {
        return joinPoint.proceed();
    }
    throw new RateLimitException(deviceId, "每分钟 1 次");
} catch (RateLimitException e) {
    throw e; // 限流异常直接抛出
} catch (Exception e) {
    // 2. Redis 不可用，降级为 Guava 本地限流
    log.warn("Redis 不可用，降级为 Guava 本地限流: {}", e.getMessage());
    RateLimiter guavaLimiter = guavaRateLimiterManager.getLimiter(deviceId);
    if (guavaLimiter.tryAcquire()) {
        return joinPoint.proceed();
    }
    throw new RateLimitException(deviceId, "每分钟 1 次");
}
```

**Guava RateLimiter 管理器（GuavaRateLimiterManager）：**

封装一个独立的组件负责管理所有设备的 Guava RateLimiter 实例，提供以下功能：

- `getLimiter(deviceId)`：获取或创建设备对应的限流器，限流速率固定为每秒 1/60 个令牌（即每分钟 1 个）。
- `cleanupInactive()`：定时清理长时间未使用的限流器实例，防止内存泄漏。建议通过 `@Scheduled` 每 10 分钟执行一次。
- 使用 `synchronized` 或 `ConcurrentHashMap` 的 `computeIfAbsent` 保证限流器创建时的线程安全。

### 6.7 优缺点分析

**优点：**

- **高可用**：Redis 宕机时自动降级为本地限流，限流功能不中断，保证系统在任何情况下都能提供限流保护。
- **算法一致**：Guava RateLimiter 和 Redisson RRateLimiter 都基于令牌桶算法，降级前后限流行为一致，不会出现限流规则突变的问题。
- **实现简单**：Guava 的 API 简洁，`RateLimiter.create()` 和 `tryAcquire()` 即可完成核心功能，集成成本低。
- **性能极高**：纯内存操作，零网络开销，纳秒级响应，降级后的性能反而优于 Redisson 方案。
- **零运维**：不依赖外部服务，不存在运维成本和可用性问题。

**缺点：**

- **分布式下不一致**：多实例部署时，各节点的 Guava RateLimiter 独立计数，降级期间可能出现某设备在实例 A 被限流但在实例 B 被放行的情况。但由于降级是临时状态（Redis 恢复后立即切回分布式限流），短暂的不一致在可接受范围内。
- **重启丢失**：JVM 重启后 Guava RateLimiter 的状态丢失，限流屏障重置。但降级期间如果发生重启，Redis 通常也是不可用状态，业务本身就不正常，限流重置的影响相对可控。
- **内存占用**：每个设备对应一个 RateLimiter 实例，占用一定的 JVM 堆内存。虽然单个 RateLimiter 实例内存占用很小（约几百字节），但如果设备规模达到数十万级别且 Redis 长时间不可用，内存累积也不可忽略。为此，方案中设计了每 10 分钟全量清理一次的定时任务。该策略的优点是实现简单、零漏删，缺点是清理后立即有上报请求时需要重建 RateLimiter，带来轻微的瞬时性能损耗。但对于正常场景（Redis 可用时降级不会触发，限流器根本不会被创建）以及降级场景（Redis 不可用时设备上报频率本身较低），这种"全量清理 + 按需重建"的机制已经足够。
- **Guava 版本兼容**：项目当前未引入 Guava，需要新增依赖，注意与项目中其他 Guava 版本的冲突问题。

### 6.8 需要引入的依赖

```xml
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>32.1.3-jre</version>
</dependency>
```

### 6.9 注意事项

- **Guava RateLimiter 的精度**：Guava RateLimiter 的 `tryAcquire()` 在调用量极低时可能存在数毫秒的精度误差，对于"每分钟 1 次"的低频率场景影响极小，可以忽略。
- **降级平滑切换**：Redisson 恢复后，AOP 切面会自动切回分布式限流（因为 try 块优先执行 Redisson），整个过程无需手动干预，对业务完全透明。
- **降级阈值设置**：建议不要在 AOP 中捕获所有异常做降级，仅捕获与 Redis 连接相关的异常（如 `RedisConnectionFailureException`），避免因其他非预期 Bug 导致误降级。
- **Guava RateLimiter 的容量**：默认的 `SmoothBursty` 模式的 bucket 容量为 `1.0`（即最多累积 1 个令牌），正好满足"每分钟 1 次"的需求，不会出现累积多个令牌导致的短时突发问题。

## 七、降级延迟问题与断路器方案

### 7.1 现象

当 Redis 挂掉时，设备上报接口的响应时间从正常的毫秒级骤升到 3~4 秒。即使代码中已经实现了 try-catch 降级到 Guava 的逻辑，每个请求仍然会卡顿数秒才会返回结果。在高并发场景下，这种延迟会导致：

- 上游调用方（如网关、API 代理）大量连接超时
- 前端请求堆积，页面长时间加载中
- 服务端线程被长时间阻塞，可能影响其他接口的正常处理

### 7.2 原因分析

延迟的根源在于 Redisson 的超时机制。当 Redis 不可用时，Redisson 不会立即抛出异常，而是需要经历以下过程：

```
try {
    // Redisson 调用（获取限流器 → 设置速率 → 获取令牌）
    RRateLimiter limiter = redissonClient.getRateLimiter(key);
    limiter.trySetRate(...);
    boolean acquired = limiter.tryAcquire();    // ← 卡在这里
} catch (RedisException e) {
    // 降级到 Guava
    fallbackToGuava(deviceId, ...);
}
```

Redisson 默认的**读取超时（timeout）为 3 秒**，即使设置了更短的超时时间，每次请求仍然要等满超时周期才能抛出异常并进入降级逻辑。具体来说：

| 步骤 | 耗时 | 说明 |
|------|------|------|
| 向 Redis 发送命令 | 约 1 秒 | TCP 连接失败 / 命令发送无响应 |
| 等待读取超时 | Redisson 默认 3 秒 | 等待 Redis 响应，直至超时 |
| 抛出 RedisException | 瞬时 | 进入 catch 块，降级 Guava |
| **合计** | **约 3~4 秒** | 每次请求都有此延迟 |

核心矛盾在于：**降级逻辑在 catch 块中，而进入 catch 的前提是等满超时**。只要 Redis 还处于不可用状态，这种"等超时→降级"的模式就会在每一个请求上重复上演。

### 7.3 解决思路对比

针对这个问题，有几种不同层次的解决思路：

**思路一：缩短超时时间**

在 `application.yml` 中调小 Redisson 的超时配置：

```yaml
spring:
  data:
    redis:
      timeout: 2000       # 从 3 秒缩到 2 秒
      connect-timeout: 2000
```

- 优点：改动最小，只需改配置
- 缺点：治标不治本。超时缩短到 2 秒，每个请求仍然要等 2 秒；如果设置太激进（如 500ms），正常场景下的网络抖动可能误触降级

**思路二：健康检查探针**

启动一个定时任务（如每 5 秒），快速 ping Redis 检查可用性：

```java
@Scheduled(fixedRate = 5000)
public void checkHealth() {
    redisTemplate.execute((RedisCallback<String>) conn -> conn.ping());
}
```

切面中直接判断标记位，Redis 不可用时跳过 Redisson 直接走 Guava。

- 优点：Redis 不可用后最慢 5 秒内，所有请求 0 延迟降级
- 缺点：引入额外的定时探测逻辑，增加少量代码复杂度；探测和实际请求之间存在窗口期（最多 5 秒的延迟）

**思路三：断路器（推荐）**

断路器的本质是"发现故障后快速切断调用链路"，与思路二的核心差异在于：

- 思路二是**主动探测**，不管有没有请求都定时去 ping
- 思路三是**基于实际调用的成败统计**，只有真正在调用时才会触发，更精确

断路器更适合"多实例部署"和"流量随机"的场景，不需要单独的探测任务，完全基于业务请求的调用结果自动决策。

**思路四：综合方案（最佳实践）**

超时配置 + 断路器 + 兜底 Guava 三层配合：

| 层 | 组件 | 职责 |
|----|------|------|
| 第一层 | 超时配置（2 秒） | 限制单次调用的最大等待时间 |
| 第二层 | 断路器 | 熔断后完全跳过 Redisson 调用 |
| 第三层 | Guava RateLimiter | 确保限流功能完全不中断 |

### 7.4 断路器方案（推荐）

#### 原理回顾

断路器的三个状态：

```
CLOSED（闭合）           OPEN（断开）           HALF_OPEN（半开）
正常调用 Redisson       熔断中，直接降级        放行试探请求
统计最近调用成败         0 延迟进入 Guava       成功→恢复，失败→继续熔断
```

关键参数：

| 参数 | 作用 | 建议值 |
|------|------|--------|
| sliding-window-size | 统计最近多少次调用的成败 | 10 次 |
| failure-rate-threshold | 失败率超过此值触发熔断 | 50% |
| wait-duration-in-open-state | 熔断持续多久后尝试恢复 | 60 秒 |
| permitted-number-of-calls-in-half-open-state | 试探请求数 | 1 次 |

#### 加入断路器后的完整降级流程

```
请求到达 → RateLimitAspect
    │
    ├── 断路器状态 == OPEN？
    │       ├── 是 → 直接走 Guava 降级（0 延迟，不调 Redis）
    │       └── 否 → 继续 ↓
    │
    ├── try { Redisson 调用 }
    │       ├── 成功 → 断路器记录成功 → 放行
    │       └── 失败（超时/异常）→ 断路器记录失败 → 抛异常
    │
    ├── catch (RateLimitException) → 直接抛出
    │
    ├── catch (RedisException)
    │       ├── 断路器检查失败率是否超过阈值
    │       │       ├── 是 → 切换为 OPEN 状态
    │       │       └── 否 → 保持不变
    │       └── 降级到 Guava 限流
    │
    └── 熔断持续时间到达 → 进入 HALF_OPEN
            ├── 放行 1 个试探请求
            ├── 成功 → CLOSED（恢复 Redisson）
            └── 失败 → OPEN（继续熔断）
```

#### 和当前方案的关键区别

对比 Redis 挂了 1 分钟（60 个请求）时的表现：

| 时间线 | 当前方案（try-catch） | 加断路器后 |
|--------|---------------------|-----------|
| 0~5 秒 | 前 2 个请求各等 2 秒降级 | 前 2 个请求各等 2 秒降级 |
| 5~7 秒 | 第 3~4 个请求各等 2 秒 | **断路器 OPEN**，第 3 个请求 0 延迟降级 |
| 7~60 秒 | 剩下 56 个请求，每个等 2 秒 | 剩下 56 个请求，全部 **0 延迟**降级 |
| **总等待时间** | **≈ 120 秒** | **≈ 4 秒**（仅前 2 次请求等待） |

#### 与健康检查探针的对比

| 对比维度 | 断路器 | 健康检查探针 |
|---------|--------|-------------|
| 触发方式 | 被动（基于请求结果） | 主动（定时探测） |
| 实现复杂度 | 中等（引入 Resilience4j） | 低（一个类 + 定时任务） |
| 是否需要额外依赖 | 需要 resilience4j | 不需要（已有 @Scheduled） |
| 探测延迟 | 取决于请求频率 | 取决于探测间隔 |
| 精确度 | 高（基于真实业务调用） | 低（探测和业务调用路径可能不一致） |
| 扩展性 | 支持多种熔断策略，与微服务体系集成好 | 仅适用单服务场景 |

#### 依赖配置

```xml
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
    <version>2.2.0</version>
</dependency>
```

```yaml
resilience4j:
  circuitbreaker:
    configs:
      default:
        sliding-window-size: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 60s
        permitted-number-of-calls-in-half-open-state: 1
```

#### 优缺点

**优点：**
- 熔断后完全跳过 Redisson 调用，降级零延迟，用户体验不受影响
- 自动恢复，不需要手动干预。Redis 恢复后自动切回分布式限流
- 基于真实调用结果统计，比定时探测更精确，不会误判
- 与微服务体系天然兼容，后续扩展为 Spring Cloud 架构时可以无缝集成

**缺点：**
- 需要额外引入 Resilience4j 依赖
- 增加了 AOP 切面的复杂度，断路器状态管理需要理解其生命周期

## 八、注意事项

### 8.1 RRateLimiter 行为说明
- `trySetRate()` 只在限流器首次创建时生效，如果限流器已存在，不会覆盖已有配置。
- 如果需修改限流配置，可以调用 `setRate()` 强制覆盖现有配置。
- `tryAcquire()` 默认立即返回结果，不会阻塞等待，适合需要快速响应的场景。

### 8.2 并发场景
- Redisson RRateLimiter 的 `tryAcquire()` 是线程安全的，多个线程同时调用时不会出现超放行。
- 多个请求同时到达同一个设备的限流器时，Redisson 内部通过 Lua 脚本保证只有一个请求能获取到令牌。

### 8.3 故障场景
- **Redis 宕机**：Redisson 默认有重试机制，如果连接失败会抛出异常。可在 AOP 切面中添加 `try-catch`，在 Redis 不可用时降级为放行（牺牲限流保可用），或拒绝所有请求（保限流但影响业务）。建议根据业务重要性选择降级策略。
- **网络抖动**：短时网络闪断时，Redisson 会自动重试，重试期间请求可能会失败或等待，建议在 AOP 中设置合理的超时时间。
- **服务重启**：限流状态存储在 Redis 中，重启后限流状态不丢失，不会出现在服务重启后设备瞬间大量请求涌入的情况。
